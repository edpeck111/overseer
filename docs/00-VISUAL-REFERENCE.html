WHAT YOU'RE LOOKING AT

A static, non-functional preview of the proposed OVERSEER v3 chrome and three example screens. Use the VIEW buttons to simulate phone landscape, tablet, and desktop widths — the same screens recompose into more panels at larger sizes. Use the SCREEN buttons to switch between HOME, COMMS (lazygit-style three-pane), and POWER (btop-style dashboard). Click : PALETTE to see the command palette overlay.

WHAT'S DEMONSTRATED

Persistent chrome — status strip top, breadcrumb second line, content middle, hotkey bar bottom. Spatial consistency: your eyes always know where to look.

Adaptive layout — the 80×40 grid is canonical; bigger viewports add side panels rather than stretching content. On phone the side stack hides; on tablet it appears; on desktop the menu goes 3-up.

256-color discipline — phosphor green dominates, amber is reserved for headers and alerts, cyan for cool/network states, red for danger. Glow on bright text via text-shadow. Subtle scanlines + vignette via fixed overlays.

Hotkey-driven — every menu item has a single accelerator letter rendered as a high-contrast key cap. The hotkey bar at the bottom mirrors the F-key pattern from Norton/Midnight Commander.

Command palette — the : overlay with fuzzy match. The escape valve for feature sprawl: top-level menus stay clean, infinity of commands lives here.

WHAT'S NOT IN THE PREVIEW (deliberately)

Live data, animations, sound, real interactivity, the other eight modules, the boot sequence, the per-module palette swaps, the typing-character-at-a-time effects, the actual functional state machine. Those all come in the build phase. This is purely about: "does the look land?"

NEXT STEPS

If this is the right direction, the handoff to Claude Code on your dev machine becomes a spec doc + this HTML as the visual reference. If anything feels off — color temperature, typography, density, the menu layout, the chrome proportions — say so and I'll iterate before you commit.
