# 05 — OMP PROTOCOL

> **Overseer Mesh Protocol** — the binary wire format between Cardputer (or any LoRa-only Overseer node) and Overseer Prime.
>
> Designed for: ≤200-byte typical packets, robust to packet loss, compressible to fit single LoRa transmissions, extensible for plugins.

---

## 1. Wire format

### 1.1 Base packet

```
 0       1       2       3       4               5+
+-------+-------+-------+-------+-------+ ─ ─ ─ +
|  ver  |  op   |     msg_id    |   payload     |
+-------+-------+-------+-------+-------+ ─ ─ ─ +
   1B      1B        2B               variable
```

| Field | Size | Description |
|---|---|---|
| `ver` | 1 byte | Protocol version. Currently `0x01`. |
| `op` | 1 byte | Opcode (see §3). |
| `msg_id` | 2 bytes | Unsigned BE. Echoed in responses. Increments per request, wraps. |
| `payload` | variable | Brotli(MessagePack(structured payload)), with shared dictionary. |

For requests, `msg_id` is set by the client. For server-pushed events, `msg_id = 0x0000`.

### 1.2 Fragmented packets

LoRa imposes a max payload (~237 bytes after Meshtastic framing). When OMP payload exceeds the available space, the packet is fragmented:

```
 0       1       2       3       4       5       6       7+
+-------+-------+-------+-------+-------+-------+-------+ ─ ─ +
|  ver  |op|MSB |     msg_id    |frag_id| total | seq#  | data |
+-------+-------+-------+-------+-------+-------+-------+ ─ ─ +
```

When the high bit of `op` is set (`op & 0x80`), the packet is a fragment.

| Field | Size | Description |
|---|---|---|
| `op|MSB` | 1 byte | High bit set indicates fragmented; lower 7 bits = real opcode |
| `frag_id` | 1 byte | Unique fragmentation session ID for this msg_id |
| `total` | 1 byte | Total fragments expected (1-255) |
| `seq#` | 1 byte | This fragment's sequence (0-indexed) |
| `data` | variable | Slice of the full payload |

Receiver buffers fragments by `(msg_id, frag_id)`, reassembles when `total` fragments received, then decodes as a normal payload.

### 1.3 Acknowledgments

Optional reliability layer. When client sends with `op = 0x01 (REQ_ACK_REQUIRED)` set as a flag in the high nibble, server replies with `0x02 (ACK)` containing the `msg_id` of the acked request.

LoRa is best-effort by default; ACKs let critical paths (message send, waypoint add) confirm.

## 2. Payload encoding

```
raw_struct → MessagePack → Brotli(dictionary=overseer.dict) → wire bytes
```

### 2.1 MessagePack profile

Standard MessagePack with these conventions:

- Maps use short-string keys (≤ 4 chars where natural)
- Timestamps as `int` Unix seconds (4 bytes), not strings
- Coordinates as `float64` pairs `[lat, lon]`
- Binary data (signatures, photos) as `bin` type
- Enums as small `int`, with mapping kept in shared schema doc

### 2.2 Shared Brotli dictionary

`overseer.dict` is a 16-32 KB binary dictionary trained on representative payloads. Generated from:

- Common JSON/MsgPack keys: `from`, `to`, `subj`, `body`, `at`, `kind`, `lat`, `lon`, `alt`, `id`, `state`, `op`, `who`, `when`, ...
- Common values: callsign patterns, board names, category names, common English words ("the", "and", "of", "to"), ANSI escape sequences (`\x1b[`...), the Overseer LLM's idiomatic phrases ("Stay sharp.", "Field medicine only.", "Cross-referencing archives...")
- A small corpus of sample messages, log entries, search results

Generated by `tools/build-dictionary.py` running over representative session captures. Versioned: dictionary `v1` ships in shell `v3.0.x`, dictionary `v2` requires bumping protocol `ver`.

### 2.3 Compression performance (target)

| Payload | Raw | MsgPack | Brotli+dict |
|---|---|---|---|
| Status ping | 120 B | 60 B | **16 B** |
| 10 inbox headers | 600 B | 340 B | **95 B** |
| Single message body 200ch | 280 B | 250 B | **70 B** |
| LLM response 500ch | 520 B | 510 B | **140 B** |
| Library article 3KB | 3000 B | 2900 B | **700 B** |
| Waypoint list (10) | 800 B | 480 B | **130 B** |

### 3.1 Protocol control (0x00-0x0F)

| Op | Name | Direction | Payload |
|---|---|---|---|
| `0x00` | HELLO | C→S | `{ver, op_id, callsign, dict_ver, caps[]}` |
| `0x01` | HELLO_ACK | S→C | `{server_ver, dict_ver, time, caps[], seq{kind→last_seq}}` |

| `0x09` | DICT_FETCH | C→S | `{}` — client requests a dict update; server streams new dict via fragments |

### 3.9 Reserved / plugin (0xF0-0xFF)

`0xF0-0xFE` reserved for plugins. Plugin manifest registers opcodes; OPi5 routes accordingly. Plugin opcodes can collide between plugins on different installs — that's fine because each Cardputer's shell knows which plugins are installed locally.

`0xFF` reserved for protocol-level extensions (future versions).

## 4. Sequence diagrams

### 4.1 Send a message (with ack)

```
Client                                Server
  |                                     |
  | OMP {0x12 MESSAGE_SEND, ack}        |
  | ────────────────────────────────────>
  |                                     | (encrypt, persist, route)
  | OMP {0x02 ACK msg_id}               |
  | <───────────────────────────────────|
  |                                     |
  | OMP {0x12 resp: {id, state, when}}  |
  | <───────────────────────────────────|
  |                                     |
  |                                     | (later, if recipient online)
  | OMP {0x0F PUSH comms.delivery       |
  |      data: {id, state: "delivered"}}|
  | <───────────────────────────────────|
```

Optimistic UI: client renders message in SENT immediately on user submit, before any of the above. UI updates from `pending → sent` on the resp, `sent → delivered` on the push.

## 5. Reliability + retransmission

LoRa is best-effort. OMP layers reliability where needed:

- **Idempotent reads** (anything starting with a `_FETCH`, `_LIST`, `_HISTORY`): no ack needed; client retries on timeout.
- **Mutating ops** (`_NEW`, `_UPDATE`, `_DELETE`, `_SEND`): ack required. Server processes once, acks. Client retries the same `msg_id` on timeout — server uses `msg_id` for dedupe.
- **Push streams** (LLM tokens, push events): if sequence gap detected, client sends `0x07 FRAG_REQ` with missing seqs; server retransmits.

Timeouts:

| Class | Default timeout | Mesh timeout |
|---|---|---|
| Status ping | 5s | 30s |
| Standard fetch | 10s | 60s |
| Send + ack | 15s | 120s |
| LLM query (start) | 10s | 30s |
| Per-token chunk | n/a | 10s gap |
| Sync hello | 30s | 180s |

## 6. Versioning

**Protocol version** in the first byte. Bumped only on breaking wire changes.

**Dictionary version** in `HELLO`. Bumped when the shared Brotli dictionary changes (which it shouldn't often). Mismatch → server replies `DICT_HASH`; client either has it cached or fetches via `DICT_FETCH`.

**Capability negotiation** in `HELLO`. Each side advertises supported caps:

```
caps: ["lora", "wifi", "vlm", "sdr-rtl", "tts-piper", "stt-whisper", ...]
```

Plugins register their own caps. Client UI greys out features not in negotiated intersection.

## 7. Security overlay

OMP itself is not encrypted at the protocol level — it relies on:

1. **Meshtastic AES at the transport layer** (channel keys; defense in depth)
2. **Application-layer crypto for sensitive content**: messages, board posts (when set to private), admin operations

Application-layer encryption format (within the MsgPack body, where applicable):

```python
{
  "ct": <bytes>,        # ciphertext (AES-256-GCM)
  "nonce": <bytes>,     # 12-byte
  "sig": <bytes>,       # ed25519 over (msg_id || ct), sender's identity key
  "key_id": <bytes>,    # which ratchet step
}
```

Public board posts: signed but not encrypted. Direct messages: encrypted to recipient's current ratchet step.

Admin opcodes (`0xA1+`) require an additional `auth_token` field obtained via `SYS_PIN_VERIFY`. Token is short-lived (15 minutes), rotated.
